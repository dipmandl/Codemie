# Feature: Add user-visible validation errors for Create/Edit Release Note form

## Story Information
- Story ID: CS-50
- Jira URL: https://mandlikdipak52.atlassian.net/browse/CS-50
- Priority: Medium
- Issue Type: Story

## Feature Overview
The release note Create/Edit form currently fails silently when required fields are missing (JS returns early). Add user-visible, accessible validation feedback.

## Business Goal
Reduce user confusion and improve data quality by ensuring users understand what is required to save a release note.

## Scope
**In scope**
- Field-level error messages for required fields (Product, Version, Title, Description, Release Date)
- Prevent create/update when invalid
- Accessibility attributes (`aria-invalid`, `aria-describedby`) and focus management
- Clear errors on correction and on successful save/cancel

**Out of scope**
- Server-side validation (no backend)
- New filters/sorting/import/export/delete

## Functional Requirements
- Validate required fields on submit for create and edit.
- If invalid: do not persist changes; show which fields are invalid.
- Mark invalid fields with `aria-invalid="true"`.
- Associate error text with inputs (e.g., `aria-describedby` referencing an error element).
- Clear field errors when user corrects input and/or after successful save.

## Non Functional Requirements
- No external dependencies.
- Validation UX must be keyboard- and screen reader-friendly.

## Existing Implementation
- `product_release_dashboard/script.js`
  - Submit handler reads `readFormValues()` and checks required fields with:
    ```js
    if (!updated.product || !updated.version || !updated.title || !updated.description || !updated.releaseDate) {
      return;
    }
    ```
  - Because `event.preventDefault()` is used, native browser `required` validation UI does not appear.
- `product_release_dashboard/index.html`
  - Inputs are marked `required` but there are no error containers.

## Proposed Changes
### 1) Add validation utilities (JS)
Add helpers:
- `validateReleaseForm(data)` → returns an object `{ fieldName: message }`.
- `clearFormErrors()` → clears messages + removes `aria-invalid` and invalid CSS class.
- `renderFormErrors(errors)` → writes messages into dedicated DOM nodes; sets aria attributes.
- `focusFirstInvalidField(errors)` → focuses the first invalid field.

Update submit handler flow:
1. `clearFormErrors()`
2. `const errors = validateReleaseForm(updated)`
3. If errors: `renderFormErrors(errors)`; `focusFirstInvalidField(errors)`; `return`
4. Else run existing create/edit logic; after saving call `clearFormErrors()`.

Add live clearing:
- Add `input` listeners to required fields that clear that field’s error once the value becomes valid.

Also update `exitEditMode()` to call `clearFormErrors()` so Cancel resets validation UI.

### 2) Add error placeholders (HTML)
For each required field, add a sibling element after the input/textarea:
- `product-error`, `version-error`, `title-error`, `description-error`, `releaseDate-error`

Example:
```html
<label>
  Product Name
  <input ... id="product" name="product" required />
  <span class="field-error" id="product-error" hidden></span>
</label>
```

Optional but recommended:
- Add a form-level summary container at the top of the form:
  - `<div id="form-error-summary" class="form-error-summary" role="alert" hidden></div>`
  - Use for a generic message like “Please fix the highlighted fields.”

### 3) Add styles (CSS)
Add styles for:
- `.field-error` (color: `--danger`, small font, spacing)
- `.invalid` on inputs/textarea/select (red border, consistent focus outline)

## File Changes
- `product_release_dashboard/script.js`
  - Replace silent return with validation + error rendering
  - Add input listeners and helper functions
- `product_release_dashboard/index.html`
  - Add error placeholder elements (and optional summary container)
- `product_release_dashboard/styles.css`
  - Add `.field-error` and `.invalid` styling

## New Files
- None

## Data Flow
Submit → readFormValues → validateReleaseForm
- If invalid: write errors to DOM + aria attributes + focus first invalid → stop
- If valid: mutate in-memory `releases` → save to localStorage → reset/edit mode → render list

## UI Changes
- Inline error messages under required fields
- Invalid fields highlighted
- Focus moves to first invalid field on failed submit

## API Changes
- None

## Database Changes
- None (localStorage only)

## Validation
- Required field presence (trimmed text fields; releaseDate non-empty)

## Error Handling
- Validation failures: no persistence; show errors.

## Edge Cases
- Edit mode + invalid submit: remain in edit mode and do not overwrite stored release.
- Cancel edit should clear validation state.

## Security Considerations
- Render error messages using `textContent` only (no `innerHTML`).

## Testing Plan
Manual:
1. Submit empty → errors for all required fields; nothing saved.
2. Fill some fields → errors persist only for missing fields.
3. Correct a field → its error clears.
4. Successful create → errors cleared; new card appears.
5. Edit existing, clear Title, save → title error shows; no persistence.
6. Cancel edit → form resets; errors cleared.

## Risks
- Layout shifts due to added error elements; mitigate via CSS spacing and `hidden` attribute.

## Dependencies
- None

## Acceptance Criteria Mapping
- AC1: invalid submit shows errors and does not save → `validateReleaseForm` + early return.
- AC2: correcting clears errors and aria-invalid updates → input listeners + clear logic.
- AC3: valid submit saves and clears all errors → `clearFormErrors` on success.

- No create/edit occurs when invalid.
- Errors clear on correction and after save/cancel.
